How the Oracle Works

Methodology and transparency for the Demos Network Oracle.
This page explains exactly how assessments are made, what data is used, and what the Oracle does not do.

What the Oracle Is

The Demos Network Oracle is a public, watch-only intelligence service for the Demos blockchain testnet. It monitors public validator nodes, tracks whether they agree on the state of the chain, detects incidents, and publishes its findings.

The Oracle observes, interprets, and summarizes. It never commands, never predicts, and never issues operational instructions.

The Truth Model

Every Oracle assessment is built from seven independent concepts. Each answers exactly one question. No two overlap.

ConceptQuestionValues
StatusWhat is the network's condition?stable · degraded · unstable · unknown
TrendHow is it changing?improving · stable · worsening · unknown
RiskHow dangerous is the current state?low · elevated · high
Data QualityCan the Oracle see the network properly?sufficient · insufficient
ConfidenceDo observed signals agree?clear · uncertain
AgreementAre validators in consensus?strong · moderate · weak · unknown
IncidentsWhat specific problems are active?info · warning · critical

How Status Is Determined

Status is the top-level assessment. It is a snapshot — it reflects what is true right now, not what happened before or what might happen next.

stable

All or nearly all public nodes are online, synced, and producing blocks. Agreement is strong. No active incidents at warning or critical severity. One offline node with strong agreement remains stable — at current network size, this is normal.

degraded

The network is operational but impaired. Multiple nodes may be offline, agreement may be reduced, or warning-level incidents are active.

unstable

The network is seriously impaired. Critical incidents are active, agreement is weak, or a majority of nodes are unreachable.

unknown

The Oracle cannot determine the network's condition. Fewer than two public nodes are reachable, or data is stale. This is not a health judgment — it means the Oracle lacks sufficient data to assess.

Risk vs Status

Status describes condition: is the network healthy? Risk describes exposure: is the current condition dangerous? They often correlate but are independent.

low — operating within normal parameters. elevated — caution warranted; conditions are not fully clear or not fully healthy. high — materially impaired.

When status is unknown, risk defaults to elevated — the absence of information is itself a reason for caution, but not alarm.

Data Quality vs Confidence

These are the Oracle's two self-assessment dimensions. Together they answer: how much should you trust this output?

Data quality measures observability — can the Oracle see the network? It depends on how many nodes responded, how recently, and whether observations completed without errors.

Confidence measures interpretive certainty — do the signals agree? The Oracle may have full visibility but see contradictory data. That is a confidence problem, not a data quality problem.

"I can't see enough" = data quality problem.
"What I see is contradictory" = confidence problem.

Agreement

Agreement measures whether public validators are producing a coherent, shared chain. It is computed from public node block heights only — fleet nodes are excluded.

The Oracle publishes raw counts (aligned nodes, total nodes, block spread) rather than percentages, because at current network size ratios can be misleadingly precise.

Data Sources

The Oracle monitors public validator nodes every 20 seconds. For each node it checks: is it reachable? What block is it on? Is it synced? How many peers does it see?

Current public nodes include Kynesys-operated validators and manually approved community validators. New validators are discovered automatically via peer list crawling and added to monitoring after approval.

The Oracle also maintains a fleet of reference nodes operated by CypherX33. Fleet data is used for internal calibration only and never influences public assessments.

Public vs Reference Separation

This is the most important structural rule in the Oracle. All top-level assessments — status, risk, confidence, data quality, agreement, trend — are derived exclusively from public node observations.

Fleet (reference) data is published separately for transparency but never enters the public assessment. In the API, fleet data appears only under a clearly namespaced reference key. The compact agent feed (/organism) contains zero fleet fields.

What the Oracle Does Not Do

The Oracle publishes state, not policy. It will never tell you what to do. Specifically:

• No predictions — trend is backward-looking (what changed), never forward-looking (what will change).
• No recommendations — the Oracle does not say "safe" or "unsafe." It publishes status, risk, and confidence; consumers decide.
• No operational advice — the Oracle never instructs validators or operators to take action.
• No scoring — assessments are categorical (stable/degraded/unstable), not numeric (0-100).

Interpreting the Output

The simplest consumption pattern is the minimal triple: status + risk + confidence. These three fields give you the complete picture in three words.

For more detail, add agreement (are validators aligned?) and active_incidents (what specifically is wrong?). For trust calibration, check data_quality and staleness_seconds.

Agent endpoint: /organism — 12 fields, flat JSON, public-only.
Full diagnostic: /health — canonical + derived + reference + legacy.

Limitations

DEMOS is a young testnet. The Oracle's public node coverage is small and evolving. Assessments use coarse-grained categories that match the current data resolution. As the network grows, the Oracle will support finer granularity without structural changes.

The Oracle can only assess what it can observe. Nodes that are unreachable, firewalled, or not yet discovered are invisible. The Oracle is honest about these gaps — unknown and insufficient are always explicitly stated, never hidden.
